# 웨딩드레스 투어 기록 앱 — 개발 스펙 v2
## 0. 이 문서를 읽는 AI에게
이 파일은 단일 소스 오브 트루스다. 좌표값, enum 값, 인코딩 규칙, 파일 구조를 임의로 바꾸지 말 것.
변경이 필요하다고 판단되면 코드를 먼저 고치지 말고 사용자에게 확인을 받은 뒤 이 문서를 먼저 수정한다.
작업은 Phase 단위로 진행한다. 한 Phase가 끝나면 멈추고 사용자 확인을 받는다. 여러 Phase를 한 번에 진행하지 않는다.

## 1. 제품 정의
**문제**: 웨딩드레스 투어 중에는 사진 촬영이 금지된다. 착용한 드레스를 기억하려면 그림을 그려야 하는데 현실적으로 불가능하다.
**해결**: 넥라인 / 실루엣 / 소매 / 텍스처 등을 탭으로 선택하면 SVG 드레스가 실시간 합성되어, 그림을 그리지 않고도 시각적 기록이 남는다.
### 사용 맥락 (설계에 직접 영향)
- 매장 안, 드레스 한 벌당 기록 시간 **15초 이내**
- 손이 자유롭지 않고 서 있는 상태, 한 손 조작
- 지하 매장이 많아 **네트워크 없음을 가정**
- 진짜 가치는 집에 돌아와서 **여러 벌을 나란히 비교할 때** 발생한다
### 확정된 요구사항
1. **일정 여유 있음** — 투어는 12월. 범위를 축소하지 않고 완성도를 올린다.
2. **범용 사용** — 제작자 본인뿐 아니라 처음 보는 예비신부가 설명 없이 쓸 수 있어야 한다.
3. **데이터 공유 필요** — 신랑·가족·친구에게 기록을 보낼 수 있어야 한다.
### 범용성이 의미하는 것 (구체적 제약)
- 드레스 용어를 모르는 사용자를 기본값으로 가정한다. **칩에 미니 SVG 썸네일을 넣고 텍스트는 보조 라벨로 둔다.** 용어를 몰라도 그림으로 고를 수 있어야 한다.
- 모든 분류에 **"기타(직접 입력)"** 탈출구가 있다. 목록에 없는 스타일 때문에 기록을 포기하는 상황이 없어야 한다.
- 매장명, 지역, 가격대 등 어떤 것도 하드코딩하지 않는다.
- 첫 실행 시 빈 화면이 아니라 **온보딩 카드 한 장**을 보여준다.
- 라벨은 `LABEL` 객체 한 곳에 모아 영어 추가가 파일 하나 수정으로 끝나게 한다. (v1에서는 한국어만 구현)
### 비목표 (하지 말 것)
- 로그인, 계정, 서버, 백엔드
- 자유 드로잉 기능
- 사실적인 드레스 렌더링 — 이건 실루엣 식별용 도식이지 일러스트가 아니다
- 매장 예약·리뷰·가격 비교 등 부가 기능

## 2. 기술 스택
- **단일 HTML 파일** (`index.html`) — 빌드 도구 없음, 번들러 없음, npm 없음
- Vanilla JS (단일 `<script>` 블록), 프레임워크 없음
- CSS는 `<style>` 인라인, 외부 CSS 프레임워크 없음
- 저장: `localStorage` (키: `dressTourData`)
- 배포: GitHub Pages (**반드시 https**, `file://` 로 열면 localStorage와 홈화면 추가가 동작하지 않는다)
- 폰트: 시스템 폰트 스택만 (오프라인 대응)
- 외부 라이브러리 없음. PNG 내보내기도 `XMLSerializer` + `canvas` 로 직접 구현한다.
### 파일 구조
```
index.html      # 앱 전부
manifest.json   # PWA 홈화면 추가용 (Phase 7)
icon-192.png
icon-512.png
SPEC.md         # 이 문서
```
Phase 6 전까지는 `index.html` 하나만 있으면 된다.

## 3. SVG 좌표계 (가장 중요)
모든 파츠는 **동일한 viewBox `0 0 400 700`** 안에서 그려진다.
아래 앵커는 상수로 선언하고 절대 하드코딩하지 않는다.
```js
const A = {
  centerX: 200,
  neckY:      105,   // 목 밑
  shoulderY:  120,   // 어깨선
  shoulderL:  140,   // 어깨점 좌
  shoulderR:  260,   // 어깨점 우
  bustY:      180,   // 가슴선
  bustL:      158,
  bustR:      242,
  waistY:     280,   // 기본 허리선 (실루엣이 덮어쓸 수 있음)
  waistL:     168,
  waistR:     232,
  hipY:       340,
  hipL:       155,
  hipR:       245,
  kneeY:      500,
  hemY:       640,   // 기본 밑단 (실루엣이 덮어쓸 수 있음)
};
```
### 3.1 파츠 분해 원칙
드레스를 통짜로 그리지 않는다. **보디스(상의)와 스커트(하의)를 허리선에서 분리**한다.
두 패스는 **좌표가 허리선에서 정확히 겹치므로 형태상 이음매가 어긋나지 않는다.**
(텍스처는 파츠별로 다를 수 있으므로 색/무늬 경계선은 보일 수 있다 — §3.3 참고)
- **보디스**: 넥라인이 결정하는 윗변 + 좌우 옆선 + 허리선
- **스커트**: 실루엣이 결정하는, 허리선에서 밑단까지의 폐곡선
- **소매**: 어깨점 기준으로 별도 부착
이 분해 덕분에 넥라인 8종 × 실루엣 6종 = 48조합이 **48개 그림 없이 14개 파츠로** 커버된다.
### 3.2 파츠 함수 시그니처
```js
// 넥라인: 보디스의 윗변 path 문자열을 반환
// 좌측 시작점 → 우측 끝점 순서로 그린다
// strapless 여부에 따라 시작/끝점이 어깨점(strap) 또는 가슴선 위(strapless)
neckline[id] = (A) => ({ d: 'M...', strapless: true|false })
// 실루엣: 허리선~밑단 폐곡선 + 이 실루엣이 쓰는 waistY 재정의
silhouette[id] = (A) => ({ d: 'M...Z', waistY: 280 })
// 소매: 어깨점 기준 좌우 대칭 path
sleeve[id] = (A) => ({ d: 'M...Z' })
```
### 3.3 렌더 순서 (z-index)
```
1. 스커트 패스     ← skirtTexture fill 적용
2. 보디스 패스     ← bodiceTexture fill 적용
3. 소매           ← sleeveTexture fill 적용
4. 외곽선 스트로크
```
텍스처는 `<defs>` 안의 `<pattern>` 으로 정의하고 `fill="url(#tex-lace)"` 로 참조한다.
텍스처마다 별도 도형을 그리지 말 것.
**스커트/보디스/소매는 각각 독립적으로 텍스처를 고른다** (예: 스커트=새틴, 보디스=비즈, 소매=오간자).
같은 파츠(스커트, 보디스)라도 서로 다른 텍스처를 쓸 수 있으므로 fill이 이제 자동으로 일치하지
않는다 — 허리 이음매는 좌표가 정확히 겹치는 것만으로 보장하고, 텍스처 경계선이 보이는 것은
정상이다(실제 드레스도 파츠마다 원단이 다르면 이음매가 보인다).
### 3.4 `other` 값 렌더링
모든 분류에 `other` 값이 존재한다. 렌더링 규칙:
- 해당 분류의 **기본값 도형을 그대로 쓰되 외곽선을 점선(`stroke-dasharray: 4 3`)으로** 바꾼다
- 사용자가 입력한 커스텀 텍스트는 그림 안에 넣지 않고 **속성 목록에만** 표시한다
- 점선은 "이 부분은 정확하지 않음"을 정직하게 알리는 신호다. 억지로 비슷한 모양을 그리려 하지 말 것
### 3.5 미니 썸네일 렌더
칩과 목록에 쓸 작은 SVG가 필요하다. 별도 도형을 만들지 않고 **동일 파츠 함수를 재사용**한다.
- 칩 썸네일: 해당 분류만 강조하고 나머지는 기본값으로 그린 뒤, `viewBox`를 해당 부위로 크롭한다
  - 넥라인 칩 → `viewBox="120 80 160 130"`
  - 실루엣 칩 → 전체, 40×70 크기로 축소
  - 소매 칩 → `viewBox="100 100 200 200"`
- 목록 썸네일: 전체 그림을 60×105로 축소
크롭 좌표도 `A` 기반 계산식으로 쓴다.

## 4. 데이터 모델
```js
// localStorage['dressTourData'] = JSON.stringify(Store)
Store = {
  version: 2,
  fittings: Fitting[],
  onboarded: boolean
}
Fitting = {
  id: string,            // crypto.randomUUID()
  shopName: string,
  visitedAt: string,     // 'YYYY-MM-DD'
  dressName: string,     // 매장 내 드레스 코드/이름, 비어도 됨
  priceKrw: number|null,
  rating: 1|2|3|4|5|null,
  memo: string,          // 자유 메모
  fitNote: string,       // 착용감 — 실제로 가장 유용한 필드
  sketch: Sketch
}
Sketch = {
  silhouette:     'aline'|'ballgown'|'mermaid'|'trumpet'|'sheath'|'empire'|'other',
  neckline:       'sweetheart'|'v'|'square'|'halter'|'offshoulder'|'bateau'|'illusion'|'straight'|'other',
  sleeve:         'none'|'cap'|'short'|'long'|'puff'|'detachable'|'other',
  waist:          'natural'|'drop'|'empire'|'basque'|'other',
  back:           'vback'|'lowback'|'keyhole'|'buttons'|'illusion'|'other',
  train:          'none'|'sweep'|'chapel'|'cathedral'|'other',
  skirtTexture:   'satin'|'mikado'|'tulle'|'lace'|'organza'|'beaded'|'other',
  bodiceTexture:  'satin'|'mikado'|'tulle'|'lace'|'organza'|'beaded'|'other',
  sleeveTexture:  'satin'|'mikado'|'tulle'|'lace'|'organza'|'beaded'|'other',
  custom:         { [category: string]: string }  // other 선택 시 사용자 입력. 예: { neckline: '아시메트릭' }
}
```
텍스처는 스커트/보디스/소매가 **각각 독립적으로** 고른다 (예: 스커트=새틴, 보디스=비즈, 소매=오간자
처럼 파츠마다 다른 원단을 쓸 수 있다). 세 필드 모두 같은 texture enum(satin/mikado/tulle/lace/
organza/beaded/other)을 공유한다.

디테일(벨트·리본 등 부가 장식) 카테고리는 두지 않는다 — 범위 밖.

**enum 순서는 인코딩에 쓰이므로 절대 재배열하지 않는다.** 값 추가는 반드시 배열 끝에만 한다.
**한글 라벨 매핑**은 별도 객체로 관리한다. enum 키에 한글을 쓰지 말 것.
```js
const LABEL = {
  ko: {
    silhouette: { aline: 'A라인', ballgown: '볼가운', mermaid: '머메이드',
                  trumpet: '트럼펫', sheath: '시스', empire: '엠파이어', other: '기타' },
    neckline:   { sweetheart: '스위트하트', v: 'V넥', square: '스퀘어넥', ... },
    texture:    { satin: '새틴', mikado: '미카도', tulle: '튤', lace: '레이스',
                  organza: '오간자', beaded: '비즈', other: '기타' },  // skirtTexture/bodiceTexture/sleeveTexture가 공유
    // ...
  }
}
```
**기본값**: 새 기록은 `aline` / `sweetheart` / `none` / `natural` / `vback` / `sweep` /
`satin`(skirtTexture) / `satin`(bodiceTexture) / `satin`(sleeveTexture)로 시작한다.
아무것도 안 골라도 저장 가능해야 한다.

## 5. 공유 (요구사항 3)
세 가지 방식을 모두 구현한다. 우선순위는 아래 순서 그대로다.
### 5.1 이미지 내보내기 — 최우선
가장 많이 쓰이는 경로다. 링크는 안 열어도 사진은 본다.
- 현재 보고 있는 화면(단일 드레스 또는 비교 3벌)을 **PNG 한 장으로** 저장
- 구현: `XMLSerializer`로 SVG 직렬화 → `data:image/svg+xml` → `Image` 로드 → `canvas.drawImage` → `canvas.toBlob`
- 이미지 안에 매장명·드레스명·별점을 텍스트로 함께 그려 넣는다. 그림만 있으면 받는 사람이 구분 못 한다
- 텍스처 `<pattern>`은 직렬화 시 `<defs>`가 함께 포함되도록 SVG 루트째로 복제한다
- 저장 후 `navigator.share`가 있으면 공유 시트를 띄우고, 없으면 다운로드로 폴백
- 출력 해상도: 실제 표시 크기의 2배 (레티나 대응)
### 5.2 URL 공유
- 스케치를 **짧은 코드**로 인코딩해 URL 해시에 담는다
- 링크를 연 사람은 **읽기 전용 뷰**로 진입한다. 기존 데이터를 절대 덮어쓰지 않는다
- 읽기 전용 뷰 하단에 "내 목록에 추가" 버튼 → 명시적으로 눌러야 저장된다
- 공유 전 **포함 항목 선택 다이얼로그**를 띄운다. 메모와 착용감은 개인적인 내용이므로 **기본 제외**
#### 스케치 코드 규격
```
형식: [버전1자] + [9개 분류 각 1자]
알파벳: '0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz' (62진)
분류 순서: silhouette, neckline, sleeve, waist, back, train, skirtTexture, bodiceTexture, sleeveTexture
각 자리 = 해당 enum 배열의 인덱스를 알파벳으로 변환
예시: "1001020154" → v1 / aline / sweetheart / cap / natural / keyhole / none /
                     skirtTexture=mikado / bodiceTexture=beaded / sleeveTexture=organza
```
- 단일 드레스 공유: `https://<host>/#d=10012032`
- 여러 벌 공유: 배열을 압축 JSON으로 만들어 base64url 인코딩 → `#s=<base64>`
  - 압축 JSON은 키를 1글자로 줄인 배열 형태 `[[code, shopName, dressName, rating], ...]`
  - **최대 10벌까지만 허용**. 초과 시 "이미지나 파일로 공유하세요" 안내
  - `other` 커스텀 텍스트는 URL에 포함하지 않는다 (길이 폭증 방지). 대신 "기타"로만 표시
디코딩 실패 시 앱이 깨지지 않고 "링크를 읽을 수 없습니다" 안내 후 일반 화면으로 진입한다.
### 5.3 JSON 백업
- **내보내기**: `Blob` + `<a download>`. 파일명 `dress-tour-YYYYMMDD.json`
- **가져오기**: `<input type="file">`. 기존 데이터와 **병합**하되 `id` 중복은 건너뛴다
- 이 두 버튼은 설정 구석이 아니라 **목록 화면 하단에 눈에 띄게** 둔다. localStorage 유실 위험 때문에 실제로 자주 눌러야 한다
- 투어 다녀온 날 백업을 유도하는 안내를 목록 화면에 한 줄 표시한다

## 6. 화면
### 6.1 온보딩 (첫 실행 1회)
카드 한 장, 3줄 이내. "매장에서 드레스를 고르고 저장하세요 / 사진 없이 기록이 남습니다 / 나중에 나란히 비교할 수 있어요". 건너뛰기 버튼 필수.
### 6.2 기록 화면 (매장에서 쓰는 화면)
레이아웃: 상단 고정 미리보기 + 하단 스크롤 선택 영역.
- 미리보기 SVG는 화면 상단 **35% 높이 고정**, 스크롤해도 항상 보인다
- 선택은 전부 **칩 버튼**. 드롭다운·슬라이더 금지
- **칩은 미니 SVG 썸네일 + 하단 텍스트 라벨** 구조 (범용성 요구사항의 핵심)
- 칩 최소 터치 영역 44×44px, 썸네일 40×40 + 라벨 11px
- 선택 즉시 미리보기 반영, **자동 저장(디바운스 300ms)**
- 매장명과 날짜는 **직전 입력값을 자동 승계** (같은 매장에서 연속 5벌을 보므로)
- "기타" 칩을 누르면 그 자리에서 인라인 텍스트 입력이 펼쳐진다
- 메모/착용감 입력란은 접혀 있고 탭하면 펼쳐진다 — 매장에서는 대개 건너뛴다
### 6.3 목록 화면
- 매장별 그룹핑, 각 항목은 미니 SVG 썸네일 + 매장명 + 별점
- 스와이프 삭제 금지 — 오작동 위험. 길게 눌러 삭제 메뉴
- 상단에 "비교하기" 토글 → 항목 선택 모드로 전환
- 하단에 백업/공유 버튼 영역
### 6.4 비교 화면
- 최대 3벌을 가로로 나란히, SVG 크게
- 하단에 속성 표: 행=속성, 열=드레스. **다른 값만 강조 표시**하고 같은 값은 흐리게
- 상단에 "이미지로 저장" 버튼
- 이게 이 앱의 존재 이유다. 여기에 가장 공을 들일 것
### 6.5 읽기 전용 공유 뷰
- URL 해시로 진입 시 표시. 상단에 "공유받은 기록입니다" 배너
- 편집 UI 없음, "내 목록에 추가" 버튼만
- 자신의 데이터가 이미 있어도 화면이 섞이지 않아야 한다
### 6.6 통계 (선택)
"8벌 중 5벌이 스위트하트" 같은 빈도 집계. 취향 파악용. 단순 리스트로 충분하다.

## 7. 시각 디자인 방향
기본값처럼 보이지 않게, 그러나 화려하지 않게.
- **팔레트**: 웨딩 = 흰색이라 배경을 흰색으로 두면 드레스가 안 보인다. 배경은 따뜻한 미드톤(예: `#EDE7E0`), 드레스는 흰색~아이보리, 강조색은 채도 낮은 세이지 또는 더스티 로즈 계열 **하나만**
- **드레스 외곽선**: 검정 아님. 배경보다 두 단계 어두운 갈색 계열 얇은 선(0.5~1px)
- **타이포**: 시스템 산세리프 한 종. 웨딩 앱이라고 필기체 쓰지 말 것 — 매장 조명 아래서 안 읽힌다
- **선택된 칩**: 색상 변화만으로 표시하지 말고 **테두리 굵기도 함께** 바꾼다 (조명 환경 편차 + 색각 접근성)
- **모션**: 미리보기 전환에 150ms 이하 페이드 외에는 애니메이션 없음
- 다크모드는 만들지 않는다. 매장 조명이 밝고, 흰 드레스는 어두운 배경에서 오히려 판단이 어렵다
- 접근성 최소선: 키보드 포커스 링 유지, 모든 아이콘 버튼에 `aria-label`, `prefers-reduced-motion` 존중

투어는 12월. **10월 말까지 완성하고 11월은 실사용 테스트와 수정에 쓴다.** 완성만 하고 실전에서 처음 쓰면 반드시 문제가 나온다.

### Phase 1 — 좌표계와 실루엣
- `index.html` 뼈대, 앵커 상수 `A`, 텍스처 없는 단색 렌더링

- 6종을 한 줄로 전부 렌더링하는 임시 확인 페이지
- **완료 조건**: 6종이 서로 명확히 구분되고, 모두 허리선 y좌표에서 정확히 시작한다
### Phase 2 — 넥라인과 합성
- 넥라인 8종 + `other` path 함수
- 보디스 + 스커트 합성 로직
- **디버그 매트릭스 페이지 필수**: 6×8 = 48조합을 격자로 전부 렌더링
- **완료 조건**: 48칸 중 이음매가 어긋나거나 형태가 무너진 칸이 0개
- 여기서 시간이 가장 많이 든다. 서두르지 말 것. 이 단계를 대충 넘기면 나중에 좌표계를 다시 손봐야 한다
### Phase 3 — 나머지 파츠
- 소매 6종, 텍스처 7종(`<pattern>`)
- 텍스처는 스커트/보디스/소매 각각에 독립적으로 적용한다 (skirtTexture/bodiceTexture/sleeveTexture)
- 각 분류의 미니 썸네일 크롭 함수
- 텍스처는 미묘하게. 레이스가 그물무늬로 보이면 실패다
### Phase 4 — 기록·저장·목록

### Phase 5 — 비교 화면
- 3벌 나란히 비교, 차이 강조 표
### Phase 6 — 공유
- PNG 내보내기 (단일 + 비교)
- 스케치 코드 인코더/디코더, URL 해시 처리, 읽기 전용 뷰
- 공유 항목 선택 다이얼로그
- **완료 조건**: 인코딩 → URL → 디코딩 왕복 테스트를 랜덤 스케치 50개로 통과

- **완료 조건**: 비행기 모드에서 앱 실행·기록·저장이 모두 동작

- 드레스 잡지나 온라인 화보를 보며 **실제로 20벌을 기록해본다.** 목록에 없는 스타일이 몇 번 나오는지 세고, 3회 이상이면 해당 분류에 값을 추가한다

- 가능하면 다른 사람 한 명에게 설명 없이 시켜보고 어디서 멈추는지 관찰한다

## 9. 코딩 규칙
- **함수는 순수하게**: 파츠 함수는 `A`를 받아 path 문자열만 반환. DOM을 만지지 않는다
- **전역 상태는 객체 하나**: `let state = { current: Fitting, view: 'record'|'list'|'compare'|'shared' }`
- 렌더는 `render()` 하나로 통일, 상태 변경 후 항상 `render()` 호출
- `innerHTML` 재생성 방식으로 충분하다. 가상 DOM 흉내 내지 말 것
- SVG path 좌표에 매직넘버 금지 — 전부 `A.xxx` 참조 또는 `A.xxx` 기반 계산식
- enum 배열 순서 변경 금지 (인코딩 호환성)
- `Store.version` 불일치 시 마이그레이션 함수를 거치게 하고, 실패해도 기존 데이터를 지우지 않는다
- 주석은 좌표 계산의 의도를 설명할 때만. 자명한 코드에 주석 달지 않는다
- 한 함수 40줄 넘으면 쪼갠다

## 10. 남은 결정 사항
- **영어 라벨을 v1에 포함할 것인가?** 범용성을 국내 사용자 범용으로 볼지, 해외까지 볼지에 따라 다르다. Phase 8에서 결정한다. 구조는 미리 잡아두므로 나중에 추가해도 비용이 크지 않다
- **뒷모습(back) 렌더링 여부**: 현재는 속성으로만 기록하고 그림에는 반영하지 않는다. Phase 5 이후 여유가 있으면 앞/뒤 전환 토글을 검토한다. 우선순위는 낮다
- **아이콘 제작**: PWA 아이콘 2종이 필요하다. Phase 7 전까지 준비
